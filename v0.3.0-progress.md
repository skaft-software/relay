## Relay v0.3.0 Progress Report

### What Was Done

#### 1. Explored the Production Relay Setup on temper-inference

**Running Docker Containers:**
- relay-gateway-1 — main relay gateway (host networking, GPU passthrough via /dev/dri)
- relay-cloud-1 — cloud API bridge proxying to DeepSeek, exposed on 127.0.0.1:1235
- suitcase-runtime-full — separate unrelated container (up 7 days)

**Systemd:** /etc/systemd/system/relay.service — a oneshot + RemainAfterExit=yes unit that just runs docker compose up -d. Enabled, active since June 15. The containers themselves have restart: on-failure:10.

**Relay config (.env):** 7 local models configured via RELAY_MODEL_MAP — qwen3.6-35b-a3b, gemma-4-26b, devstral-2-24b, glm-4.7-flash, north-mini-code, qwen3.6-27b, gemma-4-31b. Each launched via dedicated start-llama-*.sh scripts mounted read-only into the container. Eager switching policy (single GPU), 1-hour idle shutdown, prefix-cache pre-warming, circuit breaker (3 failures / 5 min window).

**GitHub releases:** v0.2.0 (latest, June 16) — Docker-first deployment, lifecycle v2, one-line install. v0.1.4 (May 12) — security hardening, thinking token budgets, surrogate-safe JSON.

#### 2. Analyzed Relay + Sandr Overlap

From the decoupling plan at docs/plans/relay-sandr-decoupling.md:
- v0.2.0 headline features (Docker-first, model lifecycle, graceful switching, orphan cleanup, idle shutdown, prefix-cache pre-warming) are ~90% Sandr-owned responsibilities per the plan
- v0.1.4 features (security, thinking tokens, JSON repair) are ~0% Sandr overlap — pure Relay API gateway work
- The decoupling plan proposed splitting relay into two packages with a provider.json file contract

#### 3. Designed the New Architecture

Key insight: Splitting relay + sandr into two packages is the wrong UX. People won't install three things. Relay should be one install with optional internal modules.

Wrote docs/plans/relay-next-release.md — the v0.3.0 plan with:
- Internal module boundaries (folders, not packages): gateway/, runtime/, cluster/, normalize/, internal/
- relay doctor CLI — one-command health check
- relay setup TUI — interactive first-time setup
- Approved model registry — curated models with VRAM estimates
- Multi-node clustering via Cloudflare Tunnel
- Cloudflare Tunnel integration for home clusters
- Phase 7: Rust rewrite — axum HTTP server, tokio process management, single static binary (~8MB), FROM scratch Docker image (~15MB), FFI to Vulkan/ROCm/CUDA

**Phase 2: relay doctor CLI** (commit 7fb0e4e)

Enhanced scripts/doctor.ts with hardware diagnostics:

- llama.cpp detection: checks /usr/local/bin/llama-server, PATH, ~/llama.cpp/build/bin/

- Network health: relay gateway, upstream, cloud bridge
- Cloud API keys: DeepSeek, Groq, OpenRouter, OpenAI, Anthropic
- Cluster status: shows mode if configured

Added CLI subcommand routing to src/main.ts:
- relay doctor — runs diagnostics, exits
- relay help — shows usage
- relay (no args) — starts server as before

Output format: clean status rows (pass/fail/skip/info) or --json for machine parsing.
